# 真實案例：診所預約系統

這是一個給非技術新手看的完整範例。你不需要看懂程式碼，只要看懂每一階段該問什麼、Codex 應該交付什麼證據。

## 場景

你想做一個小型診所預約系統。

目標：

```text
病人可以選日期和時間預約。
診所人員可以看到預約清單。
同一個時段不能被重複預約。
病人輸入錯誤資料時，系統要提示，不可以直接壞掉。
```

## 第 1 步：從 0-1 開始

使用：

```text
$nontechnical-codex-project-controller
```

貼這段給 Codex：

```text
我是非技術負責人，看不懂程式碼。
我要做一個診所預約系統。

核心目標：
1. 病人可以選日期和時間預約。
2. 診所人員可以看到預約清單。
3. 同一個時段不能被重複預約。
4. 病人輸入錯誤資料時，系統要提示。

請先不要直接寫程式。
請先幫我做現況盤點、任務分類、任務合約、成功標準、測試方式、高風險停止點、最小可執行下一步。
```

## 第 2 步：Codex 應該先產出任務合約

你應該期待 Codex 回覆類似這種白話內容：

```text
任務合約
- 目標：建立一個可預約、可查看、可防止重複預約的小系統。
- 範圍內：預約表單、預約清單、防重複、錯誤提示。
- 範圍外：付款、簡訊通知、會員登入、正式上線。
- 成功標準：可以新增預約；同一時段不能重複；錯誤輸入會提示。
- 測試方式：本機打開頁面，跑新增、重複、錯誤輸入三種測試。
- 高風險點：如果未來要收款、發簡訊、存真實個資，需要另開批准。
```

如果 Codex 一開始就直接大改很多檔案，代表流程跑太快。請要求它先補任務合約。

## 第 3 步：Codex 開始執行

你可以要求 Codex：

```text
同意這份任務合約。
請用最小可執行版本完成，完成後給我測試結果和非技術驗收腳本。
```

Codex 完成後，不要只接受「已完成」三個字。

你要看：

- 改了哪些檔案
- 怎麼啟動
- 測試有沒有跑
- 哪些還沒測
- 如果失敗會看到什麼現象

## 第 4 步：非技術驗收腳本

Codex 應該提供這種驗收方式：

```text
驗收腳本
1. 打開預約頁。
2. 輸入姓名、電話、日期、時間。
3. 按下預約。
4. 預期結果：預約清單出現這筆資料。
5. 再用同一個日期和時間預約一次。
6. 預期結果：系統顯示此時段已被預約。
7. 清空姓名後送出。
8. 預期結果：系統提醒姓名必填。
```

你不需要看程式碼，只需要照這個流程檢查畫面行為。

## 第 5 步：做到 70-90 分後，切換成稽核

使用：

```text
$nontechnical-project-readiness-auditor
```

貼這段給 Codex：

```text
這個診所預約系統目前看起來大致完成。
請不要大改功能，也不要先修 bug。
請只讀檢查目前離交付還差什麼。

請輸出：
1. 現況盤點
2. 目前分數
3. 未測項目分類
4. 高風險點清單
5. 交付結論
6. Web GPT 送審包
7. 非技術驗收腳本
8. 下一步最小收尾任務
```

## 第 6 步：稽核結果怎麼看

好結果不是「分數很高」。

好結果應該長這樣：

```text
目前分數：86/100
但分數不是交付批准。

可交付部分：
- 本機可以打開。
- 新增預約已測。
- 重複時段阻擋已測。

交付前必補：
- 錯誤輸入測試還沒跑。
- 手機畫面還沒看。
- 預約資料重整後是否保留還沒確認。

最小收尾任務：
1. 補錯誤輸入測試。
2. 補手機尺寸檢查。
3. 補一份操作說明。
```

## 第 7 步：什麼時候找 Web GPT

這種小系統不一定需要 Web GPT。

但如果你要正式給診所使用，或未來要碰到真實病人資料，就應該送外部審查。

Web GPT 應該只審查：

- 有沒有漏風險
- 有沒有漏測
- 新手 README 是否看得懂
- 是否有過度開發

Web GPT 不可以宣稱它已經跑過你的本機測試。

## 這個案例的重點

```text
0-1 用 Project Controller。
70-100 用 Readiness Auditor。
Web GPT 只做外部審查。
本機證據才是交付依據。
```
