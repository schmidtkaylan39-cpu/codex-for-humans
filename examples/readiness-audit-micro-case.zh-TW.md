# 微案例：分數高但還不能交付

這個案例示範 `nontechnical-project-readiness-auditor` 的核心觀念：

> 高分不是交付批准。交付要看證據、阻塞點與高風險狀態。

## 情境

你有一個預約系統，Codex 已經做完大部分功能：

- 使用者可以新增預約
- 後台可以看到預約列表
- 畫面可以正常打開
- 基本流程看起來可用

你問 Codex：

```text
使用 $nontechnical-project-readiness-auditor
幫我檢查這個 80-90 分專案目前離交付還差什麼。

不要大改功能。
先做：
1. 現況盤點
2. 未測項目分類
3. 高風險點清單
4. Web GPT-5.5 Pro 送審包
5. 非技術驗收腳本
6. 下一步最小收尾任務
```

## 稽核結果範例

```text
目前分數：88/100
狀態：可試用，但不能正式交付

已確認：
- 預約頁面可以開啟
- 新增預約流程有基本測試
- 後台列表可以顯示資料

未測項目：
- 重複時段是否一定被擋下：阻塞交付
- 使用者輸入錯誤電話或空白姓名時是否安全：阻塞交付
- 後台權限是否只限管理者：高風險，阻塞交付
- 通知信是否真的寄出：可由人類接受暫緩

高風險點：
- 登入權限
- 使用者資料
- 對外寄信

結論：
這個專案分數高，但仍不能正式交付，因為核心保護項目沒有 A 級證據。
下一步只需要補三個最小測試，不需要重做功能。
```

## 為什麼這很重要

非技術負責人常看到「功能大部分完成」就以為可以交付。

Readiness Auditor 的價值是把問題拆成白話：

- 哪些已經有證據
- 哪些只是看起來完成
- 哪些沒有測也可以接受
- 哪些沒測就不能交付
- 下一步最小收尾是什麼

這可以避免 Codex 一直改功能，卻沒有真正靠近交付。
